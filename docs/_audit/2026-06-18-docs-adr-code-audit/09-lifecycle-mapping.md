# Phase 2 Lifecycle Mapping (2026-06-18)

Scope: lifecycle mapping for ownership and semantic boundary audit. This file
does not move docs, rename code, change ADRs, change code, create new domains
or close open questions.

## Lifecycle Boundary Findings

| Finding | Consequence | Evidence | Confidence | Status |
| --- | --- | --- | --- | --- |
| Decisions and Obligations have candidate-first lifecycles but separate durable domains. | Task workspace can review them, but Task does not own them. | ADR-0088; ADR-0089 | high | confirmed |
| Relationship lifecycle owns trust/strength persistence when stored as relationship semantics. | Persona, Organization and Graph must not each create their own durable relationship truth. | ADR-0086; docs/domains/relationships.md | high | confirmed |
| Timeline lifecycle is rebuildable view lifecycle. | Calendar Event lifecycle remains separate from Timeline Entry lifecycle. | docs/engines/timeline.md | high | confirmed |
| Observation lifecycle is not normalized. | A generic Observations domain would be premature until concrete observation types are mapped. | docs/foundation/glossary.md; ADR-0087 | medium | open |
| Radar lifecycle is not a domain lifecycle in current docs. | Signal -> Radar -> Review -> Promote is best treated as workflow/inbox evaluation until durable Signal ownership is specified. | `rg -i "\bradar\b" docs -g '!docs/_audit/**'` found no active Radar docs; signal language appears in product/engine docs. | medium | evaluated |
