# ADR-0234: Browser-local key binding for synchronised passkeys

Статус: Принято  
Дата: 2026-07-19  
Состояние реализации: реализован owner-neutral Browser Gateway foundation:
owner-approved registration, Control Store persistence, authenticated session
proof и browser client adapter. Полный `client_gateway_v1` и public owner API
этим решением не открываются.

Зависит от:

- [ADR-0218: Owner/device identity, enrollment и offline recovery](ADR-0218-owner-device-identity-enrollment-and-offline-recovery.md);
- [ADR-0232: Browser client identity and same-origin Gateway session](ADR-0232-browser-client-device-identity-and-same-origin-session.md).

## Контекст

macOS Touch ID/iCloud Keychain может создавать backup-eligible WebAuthn
credentials. Полный запрет таких credentials делает supported first-party
browser непригодным для нормального owner flow. В то же время один
синхронизированный passkey не должен автоматически дать доступ другому Mac.

## Решение

Каждый browser device enrolment требует две независимые подписи:

1. owner-approved WebAuthn ES256 registration/assertion с required user
   verification и exact HTTPS origin/RP ID;
2. отдельный browser-local P-256 ECDSA key generated by WebCrypto.

Private browser-local key создаётся `extractable = false` и хранится browser
profile в IndexedDB как `CryptoKey`. Он не сериализуется в application state,
не передаётся Gateway, не попадает в cookie, URL, логи или diagnostics. Kernel
Control Store хранит только его 65-byte uncompressed SEC1 public point вместе
с WebAuthn credential record. Credential record также сохраняет подтверждённые
при registration `backup_eligible` и `backup_state`: эти флаги входят в
WebAuthn verification state и должны восстанавливаться без подмены defaults.

Pairing остаётся возможным только после CLI/owner-control approval. Finish
registration принимает WebAuthn attestation и exact browser-local public key
одним bounded, server-held, epoch-fenced operation. Отсутствие либо malformed
local key означает fail-closed; existing identity record без local public key
не создаёт session и должен быть paired again.

На каждом sign-in Gateway выдаёт single-use 256-bit server-held local-key
challenge вместе с WebAuthn ceremony. Browser должен вернуть WebAuthn
assertion и 64-byte raw ECDSA signature этого challenge. Gateway verifies both
before it persists assertion counter or issues the HttpOnly session cookie.
Failed local-key proof does not consume the pending ceremony.

Backup-eligible/synchronised passkey теперь допустим только в этом dual-proof
profile. Синхронизированный passkey на другом Mac не имеет его local IndexedDB
private key и поэтому не может получить session без нового CLI-approved
pairing. Device revoke and identity epoch fencing remain unchanged.

После успешной assertion Kernel атомарно сохраняет новый sign counter и
подтверждённые WebAuthn backup flags. Недопустимое состояние `backup_state =
true` при `backup_eligible = false` отклоняется contract и persistence layers.

## Последствия

- Browser identity is now the conjunction of a phishing-resistant owner
  credential and a browser-profile-local non-exportable key.
- Clearing browser profile storage intentionally removes the local key and
  requires a new owner-approved pairing; this is a fail-closed recovery path.
- Browser source may use IndexedDB only for this non-exportable `CryptoKey`
  and public credential reference, never for bearer session material.
- ADR-0232's former reject-synchronised-credential profile is superseded by
  this limited dual-proof rule.

## Проверка

The executable checks must cover success only when both proofs are valid,
rejection of a missing/invalid local-key signature, replay protection,
revocation/epoch fencing and failure of a legacy record without the local
public key. Проверки persistence также должны доказывать точное round-trip
сохранение WebAuthn backup flags. Frontend checks must prove the finish payload
contains public key or signature only, never the private key or a session
identifier.
