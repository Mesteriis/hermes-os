# Project Link Review Design

## Purpose

Hermes Hub now has a project memory spine, but project links are still derived from deterministic keyword containment. That is intentionally safe for the first slice, yet it leaves two product risks:

- broad keywords can create false project links;
- narrow keywords can miss useful project context.

This design adds a user-review workflow for project-message and project-document links without making graph edges the source of truth.

## Current Context

Relevant current decisions:

- `ADR-0001` makes meaningful state changes canonical events.
- `ADR-0015` requires explicit command/query boundaries.
- `ADR-0023` makes projections rebuildable.
- `ADR-0038` and `ADR-0040` require protected local API calls to use bearer token plus `X-Hermes-Actor-Id`.
- `ADR-0045` makes graph tables rebuildable projections.
- `ADR-0047` adds projects and keyword-derived project graph links. Those links are `suggested` until a review workflow confirms them.

Relevant current implementation:

- `backend/src/projects.rs` owns project records, keyword matching, project detail reads and derived project stats.
- `backend/src/graph_projection.rs` projects project nodes and project relationship edges from keyword matches.
- `backend/src/graph.rs` already supports `suggested`, `user_confirmed` and `user_rejected` graph review states.
- `frontend/src/lib/api.ts` and `frontend/src/routes/+page.svelte` already expose real project list/detail data.

## Approaches Considered

### Recommended: Event-Backed Review Decisions

User review commands append a canonical event, then update a durable review decision table. Projection rebuild can replay the same event into the table. Graph edges remain derived from keyword matches plus review decisions.

Tradeoff: More code than directly mutating graph edges, but it preserves ADR-0001 and ADR-0023.

### Direct Graph Edge Mutation

The UI could update `graph_edges.review_state` directly.

Tradeoff: Fast to implement, but wrong architecturally. A graph rebuild would erase the decision or require special graph state preservation.

### Keyword-Only Tuning

The UI could only edit keywords and avoid per-link review state.

Tradeoff: Useful later, but not enough. A single keyword can affect many records, and the user still needs to reject individual false positives.

## Decision

Implement event-backed project link review for direct project links to messages and documents.

The first slice reviews only these direct target kinds:

- `message`
- `document`

People and email-address project edges remain derived from accepted project-message links. This avoids creating a separate manual identity-review model inside the project workflow.

Review states:

- `suggested`: no explicit durable decision; the link is generated by keyword rules.
- `user_confirmed`: explicit user decision; the link remains active even if keyword rules later stop matching.
- `user_rejected`: explicit user decision; the link is hidden from project detail and omitted from graph projection even if keyword rules still match.

The durable review table stores only explicit decisions. Resetting a decision back to `suggested` deletes the explicit review row while still appending a canonical event for history.

## Data Model

Add `project_link_reviews`.

Logical fields:

- `project_id`
- `target_kind`
- `target_id`
- `review_state`
- `event_id`
- `actor_id`
- `reviewed_at`
- `created_at`
- `updated_at`

Uniqueness:

- one current explicit decision per `(project_id, target_kind, target_id)`.

The table does not store message or document body content.

## Event Model

Use a canonical event type:

- `project.link_review_state_changed`

Payload:

```json
{
  "project_id": "project:v1:hermes-hub",
  "target_kind": "message",
  "target_id": "message:v1:...",
  "review_state": "user_confirmed"
}
```

`review_state: "suggested"` means the explicit review decision was cleared.

The API command requires `command_id` as an idempotency key. The event source uses that key as `source.source_id`.

## API

Add protected local APIs:

- `GET /api/v2/projects/{project_id}/link-candidates`
- `PUT /api/v2/projects/{project_id}/link-reviews`

The candidate query returns direct message/document candidates with safe evidence:

- target kind and ID;
- title/subject;
- source label;
- occurrence/import timestamp;
- current review state;
- graph node ID;
- evidence excerpt limited to subject/title, never message body.

The review command sets the desired state:

- `user_confirmed`
- `user_rejected`
- `suggested`

The command validates:

- local API token;
- actor ID;
- project existence;
- target kind;
- target existence;
- command ID.

## Projection Rules

For project-message and project-document links:

- keyword match with no review decision -> active link with graph review state `suggested`;
- keyword match with `user_confirmed` -> active link with graph review state `user_confirmed`;
- no keyword match with `user_confirmed` -> active link with graph review state `user_confirmed`;
- any link with `user_rejected` -> inactive, omitted from project detail and graph edges;
- `suggested` reset -> remove explicit review decision and fall back to keyword rules.

For project-person and project-email-address links:

- derive from active project-message links only;
- use the same review state as the message link that produced the evidence;
- omit rejected messages.

## Frontend UX

Keep the existing desktop visual language. Add a compact review panel inside the Projects view:

- suggested links list;
- confirmed links visually marked;
- confirm/reject actions;
- empty state when there is nothing to review;
- error state for failed command or reload;
- no mobile-specific design or validation.

The UI must not expose raw message bodies.

## Testing

Backend:

- project link candidate query combines keyword suggestions and confirmed overrides;
- rejected direct links are excluded from project detail and graph projection;
- reset to `suggested` clears explicit decision;
- review command appends a canonical event and updates the review table;
- projection handler can rebuild the review table from `project.link_review_state_changed` events;
- API rejects missing auth, missing actor ID, invalid target kind and missing target.

Frontend:

- TypeScript check validates new API contracts.
- Build validates Svelte state and markup.
- Browser smoke validates Projects tab renders review queue and command errors do not create console errors.

## Non-Goals

- Project create/edit UI.
- Manual people/contact merge.
- Reviewing project-person edges directly.
- AI project inference.
- OCR or entity extraction.
- Mobile UI.

## Risks

- Event/table dual writes need transaction discipline. The implementation plan requires a transaction-capable event append helper.
- Existing project stats currently read from keyword matches. They must switch to active reviewed links in the same slice.
- A future keyword editor may need a separate ADR because it changes the rule source, not individual review decisions.
