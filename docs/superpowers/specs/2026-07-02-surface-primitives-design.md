# Hermes UI Surface Primitives Design

Date: 2026-07-02
Status: Draft for user review
Scope: frontend shared UI design spec only. No implementation is included in this document.

## Context

The current Hermes UI catalog already contains several surface-like primitives,
including `Surface`, `Paper`, `Panel`, `Card`, `Sheet`, `SidePanel`,
`FloatingPanel`, `InspectorPanel` and `HoverCard`. The gap is not only missing
components. The larger issue is that the surface layer is not presented as a
clear Storybook group with explicit usage rules, overlay safety rules and a
coherent component contract.

The selected direction is a generic `Hermes UI/General/Surface` pack. It must
stay domain-neutral and must not introduce Hermes domain concepts such as
evidence, persona, confidence, project or review state into shared primitive
APIs.

## Goals

- Define a first-class generic surface layer for Hermes UI.
- Normalize existing `Surface`, `Paper`, `Panel`, `Card` and `Divider` usage.
- Add missing generic primitives: `Section`, `Accordion`, `Callout`, `Well`,
  `Fieldset`, `ToolbarSection`, `StatCard` and `ActionCard`.
- Add a Storybook group named `Hermes UI/General/Surface`.
- Include explicit overlay safety coverage so dropdowns, selects, menus and
  popovers are not clipped by cards, sections or panels.
- Keep the scope generic and reusable across Hermes workspaces.

## Non-Goals

- Do not add domain components such as `EvidenceCard`, `PersonaCard`,
  `ReviewSection`, `ContextPackPanel` or `SignalCard`.
- Do not change backend APIs, state ownership, query keys or domain boundaries.
- Do not rewrite the app shell or layout architecture.
- Do not introduce new state management for surface components.
- Do not enable empty autogenerated Storybook docs pages.

## Decisions

The selected implementation approach is to normalize existing surface primitives
and add the missing generic primitives. This is intentionally broader than a
minimal patch, but narrower than a full domain composition layer.

`Section` is not a card. `Section` is a larger semantic block for pages, forms
and grouped workflow content. `Card` is for repeated objects, compact containers
and item grids.

`Accordion` belongs in the surface pack. It is a disclosure primitive for
sections, panels and cards, not a navigation tree.

`StatCard` and `ActionCard` are generic. They expose neutral fields and slots
only. Domain-specific evidence, confidence, persona, source and review concepts
must be modeled in separate domain patterns or domain components, not as fields
in these primitives.

## Component Contract

### Surface

Purpose: the lowest-level visual surface primitive.

Props:

- `as`
- `tone: default | muted | raised | deep`
- `padding: none | sm | md | lg`
- `radius: none | sm | md | lg`
- `bordered`
- `clip`

Default behavior: no clipping. `clip` must be explicit.

### Paper

Purpose: an opinionated content or reading block over `Surface`.

Props should align with `Surface` where useful:

- `as`
- `tone`
- `padding`
- `radius`
- `bordered`
- `clip`

Paper should not own complex header or action slots.

### Panel

Purpose: workspace pane, inspector pane or application-level surface.

Props should align with `Surface` where useful:

- `as`
- `tone`
- `padding`
- `radius`
- `bordered`
- `clip`

Panel is more structural than `Paper`, but it must remain domain-neutral.

### Card

Purpose: repeated item, compact object container or grid item.

Props:

- `as`
- `variant: default | muted | raised | interactive`
- `density: compact | comfortable`
- `selected`
- `disabled`
- `clip`

Slots:

- `header`
- default
- `footer`
- `actions`

Existing subcomponents remain part of the public API:

- `CardHeader`
- `CardTitle`
- `CardDescription`
- `CardContent`
- `CardFooter`

### Section

Purpose: larger semantic block in a page, form or workspace flow.

Props:

- `as`
- `title`
- `description`
- `eyebrow`
- `tone`
- `density`
- `divided`
- `clip`

Slots:

- `header`
- `actions`
- default
- `footer`

`Section` should not look like a floating card by default.

### Accordion

Purpose: disclosure within surfaces.

Props:

- `items`
- `modelValue`
- `type: single | multiple`
- `tone: plain | contained`

Slots should allow item header and item content customization. Accordion should
not become a navigation tree.

### Callout

Purpose: visible contextual message or status block.

Props:

- `tone: info | success | warning | danger | neutral`
- `title`
- `icon`

Dismissible behavior is out of scope for the first implementation slice.

### Well

Purpose: quiet inset or nested context block.

Props:

- `tone: neutral | muted | inset`
- `clip`

Well is for secondary context, not status messaging.

### Fieldset

Purpose: accessible grouping for form fields.

Props:

- `legend`
- `description`
- `required`
- `disabled`

Default slot contains form controls.

### ToolbarSection

Purpose: compact grouping inside a toolbar or command row.

Props:

- `label`
- `density`
- `align`

ToolbarSection must not own toolbar state.

### StatCard

Purpose: generic metric card.

Props:

- `label`
- `value`
- `description`
- `trend`
- `icon`
- `tone`

Slots:

- `meta`
- `footer`

### ActionCard

Purpose: generic call-to-action card.

Props:

- `title`
- `description`
- `icon`
- `tone`
- `disabled`

Slots:

- `actions`
- `meta`
- `footer`

## Styling Contract

The surface pack uses stable classes and token-driven CSS:

- `hermes-surface`
- `hermes-paper`
- `hermes-panel`
- `hermes-card`
- `hermes-section`
- `hermes-accordion`
- `hermes-callout`
- `hermes-well`
- `hermes-fieldset`
- `hermes-toolbar-section`
- `hermes-stat-card`
- `hermes-action-card`

Templates should not become Tailwind-heavy. Tailwind may be used in Storybook
outer layout where existing stories already do that, but UI kit component
styling should live in CSS and design tokens.

Interactive cards need visible `focus-visible`, hover, disabled and selected
states.

## Overlay Safety Contract

Surface primitives must not clip overlays by default.

Default rule:

- `Card`, `Section`, `Panel`, `Paper`, `Surface` and `Accordion` use
  `overflow: visible` unless `clip` is explicitly enabled.

Opt-in clipping:

- `clip` is allowed only for explicit visual clipping cases such as media
  previews, skeleton masks or decorative backgrounds.
- `clip` must be visible in the component API and in Storybook examples.

Overlay rule:

- `DropdownMenu`, `Select`, `SearchableSelect`, `MultiSelect`, `Menu`,
  `ContextMenu`, `Popover` and similar overlay primitives must render through
  existing portal or overlay-host behavior when needed.
- Overlays must be able to escape the visual bounds of cards, sections and
  panels.

Storybook must include a dedicated `Overlay Safety` story showing overlays
inside `Card`, `Section` and `Panel` without clipping.

## Storybook Plan

Add one Storybook group:

```text
Hermes UI/General/Surface
```

Stories:

- `Overview`: matrix of `Surface`, `Paper`, `Panel`, `Card` and `Section`.
- `Cards`: default card, interactive card, selected card, disabled card,
  `StatCard` and `ActionCard`.
- `Sections`: page and form sections with actions, footer and dividers.
- `Accordion`: single and multiple modes, plain and contained tones.
- `Callouts And Wells`: callout tones and nested well examples.
- `Fieldset And Toolbar`: field grouping and compact toolbar sections.
- `Overlay Safety`: dropdown, select or popover inside cards and sections,
  proving overlays are not clipped.

All stories must support Storybook locales `ru`, `en` and `es`.

The Storybook coverage guard must require `Hermes UI/General/Surface`.

## Validation Plan

Targeted validation:

- Public exports include all surface components.
- Storybook coverage requires `Hermes UI/General/Surface`.
- Surface components default to the non-clipping contract where practical.
- Storybook includes an `Overlay Safety` story.

## Risks

- The existing `Card`, `Surface`, `Paper` and `Panel` components may already
  have inconsistent styling assumptions. Implementation should normalize them
  carefully without breaking current stories.
- `clip` is useful but risky. It must remain opt-in and visible in examples.
- Visual regression is already expensive. Adding surface stories increases the
  screenshot count and runtime.
- Overlay safety depends on existing overlay primitives using portal behavior
  consistently. If a specific overlay component does not portal correctly, that
  should be fixed at the overlay primitive boundary rather than by weakening
  surface defaults.

## Open User-Approved Decisions

- Approach: normalize existing surface primitives and add missing generic
  primitives.
- Scope: extended surface pack.
- `Section` is not `Card`.
- `Accordion` belongs in the surface pack.
- `StatCard` and `ActionCard` are generic.
- Storybook uses one group: `Hermes UI/General/Surface`.
- Surface clipping is opt-in via `clip`; default is non-clipping.
