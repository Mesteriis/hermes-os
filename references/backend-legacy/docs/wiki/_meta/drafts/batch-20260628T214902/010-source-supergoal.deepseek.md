## repo-state.sh

`repo-state.sh` — Bash-скрипт, оценивающий **полное состояние рабочего дерева** относительно заданного коммита-базиса. Он применяется в финальном аудите и пофазовых проверках чистоты, чтобы гарантировать, что ни одно изменение не осталось незамеченным, даже если автономный запуск `supergoal` не делал коммитов.

**Путь в репозитории**: `.supergoal/hermes-frontend-migration-vue-3-WzENWm/repo-state.sh`

### Проблема, которую решает скрипт

Обычная команда `git diff <baseline>..HEAD` сравнивает только два коммита. Если работа не была закоммичена, diff оказывается пустым — все написанные файлы выглядят как «отсутствующие», а поиск отладочных строк через `grep` возвращает ноль совпадений, сколько бы их ни было в реальности. Скрипт исправляет эту ситуацию.

### Стратегия: полное состояние против базиса

Скрипт включает в сравнение:

- **Отслеживаемые изменения** — закоммиченные, добавленные в индекс, неиндексированные и удалённые файлы.
  Для их обнаружения используется `git diff <baseline>` (не `..HEAD`), который сравнивает **рабочее дерево** с базовым коммитом.
- **Неотслеживаемые файлы** — новые файлы, никогда не попадавшие в индекс. Они невидимы для `git diff`,
  поэтому выявляются отдельно через `git ls-files --others --exclude-standard`.

Скрипт не модифицирует репозиторий или индекс. Весь вывод предназначен для транскрипта аудита.

### Подкоманды

```bash
repo-state.sh deliverable   <baseline> <path>   # present|missing (+ подтверждение), код 0|1
repo-state.sh changed-files <baseline>          # список путей, изменённых с базиса
repo-state.sh added-lines   <baseline>          # все новые строки с базиса (для grep)
```

#### deliverable

Проверяет, существует ли заданный артефакт (файл) в текущем состоянии проекта. Логика:

1. Если файл **изменился** относительно базиса (закоммичен, staged, unstaged или удалён) — выводится `present — changed vs baseline` с краткой статистикой из `git diff --stat`.
2. Если файл — **новый неотслеживаемый** — выводится `present — untracked new file`.
3. Если файл существует на диске или отслеживается Git, но не изменился с базиса — выводится `present — exists, unchanged since baseline`.
4. Иначе — `missing`, код возврата 1.

Когда базис недоступен (не git-репозиторий, невалидный SHA, sentinel `no-git`), подкоманда переключается на простую проверку существования в файловой системе.

#### changed-files

Возвращает объединённый отсортированный список путей всех файлов, появившихся или изменившихся с момента базиса:
- вывод `git diff --name-only <baseline>` (изменённые/staged/удалённые),
- вывод `git ls-files --others --exclude-standard` (неотслеживаемые).

При недоступном базисе вывод пуст.

#### added-lines

Генерирует поток **новых строк**, предназначенный для передачи в `grep` (например, для поиска забытых отладочных операторов при проверке чистоты кода):

- строки с префиксом `+` из `git diff <baseline>` (исключая заголовки `+++`),
- полное тело каждого неотслеживаемого **текстового** файла (каждая строка считается новой).

Бинарные файлы отфильтровываются (`LC_ALL=C grep -Iq .`). Имена файлов с пробелами обрабатываются корректно благодаря ключу `-z` в `git ls-files`.

### Детали реализации

- Проверка нахождения внутри git-репозитория: `git rev-parse --is-inside-work-tree`.
- Валидация базиса: `git rev-parse --verify --quiet "${b}^{commit}"`.
- Режим строгой обработки ошибок: `set -uo pipefail`.
- Зависимости: `bash`, `git`, `grep`, `sed`, `cat`, стандартные утилиты POSIX.
- Пути с пробелами поддерживаются при условии, что вызывающая сторона корректно закавычивает параметры.
